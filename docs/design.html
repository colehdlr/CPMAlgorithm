1. Overview

This project computes and visualizes the Critical Path Method (CPM) schedule for a fixed website-development plan. Activity durations are deterministic integers — there is no PERT variance, no probabilistic estimation, and no optimistic/pessimistic fields. The tool loads the schedule from JSON, runs the forward/backward CPM passes, prints a report, and draws an interactive network diagram (with an optional Gantt view) using raylib.

Goals
  • Deterministic, unit-testable CPM core with zero I/O dependencies.
  • Clean module boundaries so four developers can work in parallel.
  • Headless mode for CI and scripted use; raylib rendering for interactive exploration.
  • Known-answer regression test: project duration = 22, critical path = A→C→E→F→G.
Non-goals
  • Variable/stochastic durations, resource leveling, or cost analysis.
  • Editing the schedule in the UI (read-only visualization).

5. Module Architecture

Five translation units with strict, one-directional dependencies. The CPM core depends only on the graph types; rendering and I/O are leaves that consume immutable results. This keeps the algorithm pure and testable.

6. Developer Responsibilities

Dev 1 — Data & Graph
graph.h · graph.c
  • Core types Activity { id, name, duration, deps[] } and Graph { activities[], n }.
  • Parse activities.json (hand-rolled or cJSON) into a Graph.
  • Build predecessor + successor adjacency lists.
  • Validate: missing dep ids, duplicate ids, cycle detection.
  • Provide topological sort (Dev 2 depends on it).
  • Durations stored as int — no variance/PERT fields.
Dev 2 — CPM Algorithm
cpm.h · cpm.c
  • Pure computation over a Graph → CpmResult.
  • Forward pass (topo order) → ES/EF.
  • Backward pass (reverse topo) → LS/LF.
  • slack = LS − ES; critical when slack == 0.
  • Emits critical-path id list + total duration.
  • No I/O, no raylib — trivially unit-testable.
Dev 3 — raylib Rendering
render.h · render.c
  • Owns window, draw loop, all raylib calls; links -lraylib (+ macOS frameworks).
  • Layered layout: x from topo depth/ES, y by lane.
  • Draw nodes (id, name, duration, 4 CPM values) and edges with arrowheads.
  • Highlight critical nodes/edges in a distinct color.
  • Stretch: pan/zoom, Gantt toggle, hover tooltips.
  • Consumes Graph + CpmResult; never mutates them.
Dev 4 — Shell, I/O, Build & Tests
main.c · io.h/.c · tests/ · Makefile
  • Wires load_graph → run_cpm → print_report / render_window.
  • CLI: input path, --headless, --export csv|dot.
  • Pretty-print CPM table to stdout.
  • Known-answer tests (duration 22, path A→C→E→F→G).
  • Maintains Makefile, CI, and README.

7. Module Interfaces

graph.h (Dev 1)

#define MAX_ID    8
#define MAX_NAME  64
#define MAX_DEPS  16
#define MAX_ACTS  64

typedef struct {
    char id[MAX_ID];
    char name[MAX_NAME];
    int  duration;                 /* fixed integer days */
    char deps[MAX_DEPS][MAX_ID];   /* predecessor ids    */
    int  ndeps;
} Activity;

typedef struct {
    Activity acts[MAX_ACTS];
    int n;
    /* adjacency by index, filled during load */
    int  succ[MAX_ACTS][MAX_ACTS]; int nsucc[MAX_ACTS];
    int  pred[MAX_ACTS][MAX_ACTS]; int npred[MAX_ACTS];
} Graph;

typedef enum { GRAPH_OK, GRAPH_ERR_IO, GRAPH_ERR_PARSE,
               GRAPH_ERR_DUP_ID, GRAPH_ERR_MISSING_DEP,
               GRAPH_ERR_CYCLE } GraphStatus;

GraphStatus load_graph(const char *path, Graph *out);
int         graph_index_of(const Graph *g, const char *id);   /* -1 if absent */
GraphStatus graph_validate(const Graph *g);
/* writes topo order (indices) into order[]; returns count or -1 on cycle */
int         graph_toposort(const Graph *g, int order[]);

cpm.h (Dev 2)

typedef struct {
    int es, ef, ls, lf, slack;   /* per activity (by index) */
    int critical;                /* 1 if slack == 0         */
} ActivityTimes;

typedef struct {
    ActivityTimes t[MAX_ACTS];
    int  n;
    int  project_duration;
    int  critical_path[MAX_ACTS];   /* ordered indices */
    int  ncritical;
} CpmResult;

/* Pure: no I/O. Returns 0 on success, -1 on cycle/invalid graph. */
int run_cpm(const Graph *g, CpmResult *out);

io.h (Dev 4) & render.h (Dev 3)

/* io.h */
typedef struct {
    const char *input_path;
    int  headless;          /* skip window  */
    enum { EXP_NONE, EXP_CSV,
           EXP_DOT } export; /* --export     */
} CliOptions;

int  parse_args(int argc, char **argv,
                CliOptions *o);
void print_report(const Graph *g,
                  const CpmResult *r);
int  export_csv(const Graph *g,
                const CpmResult *r, FILE *f);
int  export_dot(const Graph *g,
                const CpmResult *r, FILE *f);
/* render.h */
/* Blocks running the raylib draw loop until
   the user closes the window. Read-only on
   both inputs. No-op in headless mode. */
void render_window(const Graph *g,
                   const CpmResult *r);

10. Risks & Decisions

  • Fixed durations only. No PERT/variance fields — keeps the model and JSON schema simple, per requirements.
  • raylib as a hard dep. Mitigated by --headless and a no-raylib CI build so the CPM core and tests never need a display.
  • Purity of the core. cpm.c takes const Graph* and writes only CpmResult — enables deterministic unit tests and decouples Dev 2 from Dev 3.
  • Fixed-size arrays. MAX_ACTS=64 is ample for this scenario; revisit with dynamic allocation only if larger schedules are needed.
  • JSON parsing. Hand-rolled parser is acceptable for the fixed schema; cJSON is the drop-in upgrade if the schema grows.
